O diagnóstico central: o app converte rápido, mas não retém

49% dos users compram no primeiro dia — a paywall funciona. Mas 76% nunca renovam e 43.8% dos ativos são zombies (não abrem há 30+ dias). O modelo está gerando receita pontual, não recorrente. A maior ameaça não é aquisição — é o churn silencioso que vai corroer o MRR quando os anuais expirarem.

Receita inflada: $3.2K/mês vem de quem não usa o app

41.6% do MRR vem de zombies — people paying but not using. Isso mascarará métricas por meses (especialmente nos anuais que expiram entre Jul-Nov 2026), mas quando chegar a hora de renovar, a taxa de churn real vai aparecer. Esse é o risco invisível #1.

O dinheiro escondido: reengajamento de 1.160 zombies ativos

Se você conseguir reativar 30% dos zombies antes da expiração, são ~348 renovações preservadas. A US$18/ano, isso são ~$6.2K salvos. Push notifications, emails de reengajamento e deep links pro conteúdo mais recente são as alavancas.

Recomendações Priorizadas AÇÕES
Diagnóstico final: O modelo é frágil

O Heat Game tem boa aquisição e conversão, mas retenção crítica. 76% nunca renovam, 43.8% dos ativos são zombies, e o LTV por coorte está caindo 60%. O modelo atual sobrevive de novos anuais, mas o pico de expiração (Jul-Nov 2026) vai expor a fragilidade. O foco dos próximos 90 dias precisa ser retenção, não aquisição.

IMEDIATO (Próximas 2 semanas)

  1. Campanha de reengajamento segmentada para 1.160 zombies ativos
    Push notifications + email + deep link. 3 tiers: 30-60d (suave), 60-90d (urgência), 90d+ (last chance). Target: reativar 30% = ~$6.2K/ano preservado.
  2. Auditar paywall Android — mensal não está convertendo
    Apenas 4 subscribers monthly no Android (0.7%). Verificar se o plano mensal está visível na paywall Android/Stripe. Potencial: 500+ conversões adicionais.
  3. Implementar onboarding pós-purchase
    49% compram no dia 1 mas param de usar. Criar sequência de 7 dias pós-compra (email/push) mostrando features key do app. O objetivo é transformar "compra impulsiva" em "hábito".

CURTO PRAZO (30-60 dias)

  1. Redesenhar retenção: introduzir loops de engajamento no produto
    Desafios diários, streaks, notificações de conteúdo novo, ranking social. O produto precisa de motivos para o user voltar todo dia. Power users (89) fazem 3.1 renewals — estudar o que fazem de diferente.
  2. Preparar para o pico de expiração Jul-Nov 2026
    ~2.400 anuais expiram nesse período. Criar campanha de renovação antecipada (desconto de 20-30% para renovar antes de expirar). Win-back flows automatizados no RevenueCat.
  3. Testar pricing: o anual a $18 pode estar subprecificado para mercados internacionais
    Portugal ($35 ticket), US ($28), GB ($33) pagam significativamente mais. Testar pricing regional: $24.99/ano para US/EU, manter $17.99 para BR.

MÉDIO PRAZO (90 dias)

  1. Implementar Scheduled Data Exports no RevenueCat
    Dados mais completos que o CSV de 4.5K linhas. Conectar a um data warehouse (BigQuery free tier) para análises recorrentes e automáticas.
  2. Configurar attribution (media_source está vazio)
    Zero dados de canal. Sem isso, não dá pra saber quais canais trazem bons users vs zombies. Integrar AppsFlyer/Adjust com RevenueCat para LTV por canal.
O app tem product-market fit para conversão

49% convertem no dia 1, 34.8% em menos de 1 hora. A paywall funciona. O pricing é aceitável. O problema não é converter — é reter e criar valor recorrente. Se você resolver a retenção, o negócio escala. É isso que separa um app de $90K/ano de um de $500K+.
